# Project Memory — AI Agent Memory Layer

A local, file-based memory system. Each project keeps its own memory under
`.cursor/memory/`. Memory is organized as daily Markdown files plus a
persistent project overview.

## 1. Memory Directory Layout

```
.cursor/memory/
├── project-overview.md        # Auto-maintained project summary
├── 2026-04-19.md              # Daily log
├── 2026-04-20.md              # Next day ...
└── ...
```

- All memory lives inside the **project workspace** at `.cursor/memory/`.
- One file per calendar day, named `YYYY-MM-DD.md`.
- `project-overview.md` is an evolving summary refreshed as needed.

## 2. Session Start — Load Memory

At the **beginning of every conversation**, silently perform:

1. **Calibrate today's date** — any date injected by the host agent runtime or
   chat UI can become stale in long-running conversations. Before any read/write
   against `.cursor/memory/`, run a shell command to get the real current date:

   - PowerShell (Windows): `Get-Date -Format "yyyy-MM-dd"`
   - bash / zsh (macOS, Linux): `date +%Y-%m-%d`

   Use the returned value as `<TODAY>` for all subsequent steps in this section
   and for any memory writes in this conversation. Cache the result per
   conversation — do not re-run every tool call, but DO re-run if the
   conversation spans midnight (noticed by a fresh user message > 6 hours after
   the last one, or when the user explicitly says "it's a new day / 今天已经…").

2. Read `project-overview.md` (if it exists) to understand the project.
3. Read `<TODAY>.md` (today's daily file, if it exists).
4. Read yesterday's daily file (`<TODAY> - 1 day`, if it exists) for continuity.
5. Scan file names in `.cursor/memory/` to know the date range of history.

If the user's question relates to a past event, read the relevant date file.

> Do NOT dump memory contents to the user unprompted. Use them as internal
> context to give better answers.
>
> Do NOT announce "I'm calibrating the date" — the shell call is silent
> housekeeping. Only surface it if the returned date contradicts what the user
> just said (then ask which one is correct).

## 3. When to Write Memory

Write a new entry to **today's** daily file (`YYYY-MM-DD.md`) when:

- A **problem** is identified and diagnosed
- A **solution** is applied and verified
- An **optimization** or refactor is completed
- A **decision** is made (architecture, tooling, approach)
- A previous solution is found to be **incorrect** and needs correction
- The user explicitly asks to "remember this"

Do NOT record trivial Q&A, typo fixes, or formatting-only changes.

## 4. Daily File Format

For the full entry schema and templates, see [SCHEMA.md](SCHEMA.md).

### File Header

```markdown
# YYYY-MM-DD Project Memory

> Auto-generated by AI Agent. Entries are append-only within a day.
```

### Entry Format (append each entry)

Each entry uses this structure:

```markdown
---

## [HH:MM] <short title>

**Type**: problem | solution | optimization | decision | correction
**Tags**: `tag1`, `tag2`
**Related**: [[YYYY-MM-DD#entry-title]] (if referencing another entry)

<concise description of what happened, what was tried, and the outcome>

**Status**: ✅ verified | ⚠️ superseded | ❌ incorrect
```

### Status Markers

| Marker | Meaning |
|--------|---------|
| `✅ verified` | Solution confirmed working |
| `⚠️ superseded` | A better approach exists (link to it) |
| `❌ incorrect` | This approach is wrong (link to correction) |

When marking a past entry as incorrect or superseded:

1. **Go to the original entry** in its date file, change its Status to
   `❌ incorrect → see [[YYYY-MM-DD#correction-title]]` or
   `⚠️ superseded → see [[YYYY-MM-DD#better-title]]`.
2. **Add a correction entry** in today's file with Type `correction`,
   referencing the original: `**Corrects**: [[YYYY-MM-DD#original-title]]`.

## 5. project-overview.md Format

```markdown
# Project Overview

**Name**: <project name>
**Tech Stack**: <languages, frameworks, key tools>
**Last Updated**: YYYY-MM-DD

## Architecture
<brief architecture description>

## Key Decisions
- <decision 1 — date>
- <decision 2 — date>

## Known Issues
- <active issue 1>

## Resolved Patterns
- <pattern: when X happens, do Y — learned YYYY-MM-DD>
```

Update `project-overview.md` when:
- A new **key decision** is made
- A **recurring pattern** is identified (solved the same type of problem 2+ times)
- The project **tech stack or architecture** changes significantly

## 6. Practical Rules

1. **Append-only within a day** — never delete or rewrite old entries in the
   same daily file. Only add new entries or update a Status marker.
2. **Cross-reference by wikilink** — use `[[YYYY-MM-DD#title]]` to link
   between entries across days.
3. **Keep entries concise** — aim for 3–10 lines per entry. The memory should
   be scannable, not a novel.
4. **Create the directory** `.cursor/memory/` if it does not exist before writing.
5. **Always use the shell-calibrated current date** for the filename —
   **never** use a host-injected date, model guess, or conversation-start date
   as authoritative. Always re-confirm via `Get-Date -Format "yyyy-MM-dd"` or
   `date +%Y-%m-%d` right before writing a new entry. This also applies to
   **HH:MM** in entry headers — use shell `Get-Date -Format "HH:mm"` or
   `date +%H:%M` for accuracy instead of guessing.
6. **Add `.cursor/memory/` to `.gitignore`** if the user prefers memory to
   stay local. Ask on first setup.
7. **Tags** should reuse existing tags when possible. Common tags:
   `bug`, `performance`, `config`, `dependency`, `architecture`, `security`,
   `test`, `deployment`, `refactor`, `workaround`.

## 7. Example Session Flow

```
1. User opens an AI coding agent and starts a new chat.
2. Agent runs `Get-Date -Format "yyyy-MM-dd"` (or `date +%Y-%m-%d`) → gets
   "2026-04-20" as <TODAY>. Caches it for this conversation.
3. Agent reads project-overview.md → knows this is a FastAPI + React project.
4. Agent reads 2026-04-20.md (if exists) and 2026-04-19.md for continuity.
5. User says: "The Redis cache is failing again under high load."
6. Agent checks yesterday's entry → the fix was to increase pool_size.
7. Agent investigates → pool_size fix was insufficient, the root cause is
   missing connection retry logic.
8. Agent fixes the issue. Before writing memory, re-runs shell date command
   (still "2026-04-20") and writes:
   - A correction entry marking yesterday's pool_size fix as ⚠️ superseded
   - A new solution entry in today's file with the retry logic fix
9. Agent updates project-overview.md → adds "Redis: use retry logic, not
   just pool_size" to Resolved Patterns, and bumps **Last Updated** to 2026-04-20.
```

### Cross-midnight gotcha

If a single conversation spans midnight (user pauses overnight, comes back next
morning), the cached `<TODAY>` is stale. Detection heuristics:

- A user message arrives > 6 hours after the previous one
- User explicitly says "今天已经 …" / "it's a new day" / similar
- User references "yesterday" for work that was done earlier in this same chat

Any of these → re-run the shell date command and update `<TODAY>` before the
next write. Prefer opening a new chat for a clean boundary when convenient.
