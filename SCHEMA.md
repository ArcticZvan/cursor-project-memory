# Memory Entry Schema & Templates

## Daily File Template

When creating a new daily file `YYYY-MM-DD.md`:

```markdown
# YYYY-MM-DD Project Memory

> Auto-generated by Cursor Agent. Entries are append-only within a day.
```

## Entry Templates

### Problem

```markdown
---

## [HH:MM] <short problem description>

**Type**: problem
**Tags**: `tag1`, `tag2`

<What went wrong. Include error messages, symptoms, or unexpected behavior.>

**Status**: ✅ verified
```

### Solution

```markdown
---

## [HH:MM] Fix: <short solution description>

**Type**: solution
**Tags**: `tag1`, `tag2`
**Related**: [[YYYY-MM-DD#problem-title]]

<What was done to fix it. Include the key code change, config change, or
command that resolved the issue. Keep it actionable — someone reading this
in the future should be able to reproduce the fix.>

**Status**: ✅ verified
```

### Optimization

```markdown
---

## [HH:MM] Optimize: <what was improved>

**Type**: optimization
**Tags**: `tag1`, `tag2`

<What was optimized and why. Include before/after metrics if available
(e.g. response time, build time, memory usage, accuracy).>

**Before**: <metric or behavior>
**After**: <metric or behavior>

**Status**: ✅ verified
```

### Decision

```markdown
---

## [HH:MM] Decision: <what was decided>

**Type**: decision
**Tags**: `tag1`, `tag2`

<What was decided and the reasoning. List alternatives considered and why
they were rejected.>

**Alternatives considered**:
- <option A> — rejected because ...
- <option B> — rejected because ...

**Status**: ✅ verified
```

### Correction (marking a past entry as wrong)

```markdown
---

## [HH:MM] Correction: <what was wrong and the real fix>

**Type**: correction
**Tags**: `tag1`, `tag2`
**Corrects**: [[YYYY-MM-DD#original-entry-title]]

<Why the original approach was wrong. What the correct solution is.>

**Status**: ✅ verified
```

When writing a correction, also update the **original entry's Status** to:
```
**Status**: ❌ incorrect → see [[YYYY-MM-DD#correction-entry-title]]
```
or:
```
**Status**: ⚠️ superseded → see [[YYYY-MM-DD#better-entry-title]]
```

## Tag Conventions

Reuse these common tags before inventing new ones:

| Tag | Use for |
|-----|---------|
| `bug` | Runtime errors, unexpected behavior |
| `performance` | Speed, memory, resource usage |
| `config` | Configuration files, environment variables |
| `dependency` | Package versions, compatibility |
| `architecture` | System design, module structure |
| `security` | Auth, permissions, vulnerabilities |
| `test` | Test failures, test strategy |
| `deployment` | CI/CD, Docker, hosting |
| `refactor` | Code restructuring without behavior change |
| `workaround` | Temporary fix, not ideal |
| `data` | Data format, migration, schema |
| `ui` | Frontend, styling, layout |
| `api` | Endpoints, request/response |

## Cross-Reference Syntax

Use wikilink style: `[[YYYY-MM-DD#entry-title]]`

- The `entry-title` is the heading text lowercased with spaces replaced by hyphens.
- Example: `## [14:30] Fix: Redis retry logic` → `[[2026-04-20#fix-redis-retry-logic]]`
